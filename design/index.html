I Rewrote My Entire Robotics Stack in Rust

The GC pauses were fine — until I strapped a six-axis arm to a Raspberry Pi 5 running real-time inference. Here's the 11-week journey from Go to Rust, what broke, and what got faster.

KT
Kauti / Apr 20, 2026 / 18 min read
Robotics Rust Real-time Embedded Performance Control Systems
Contents

It started with a crash. Not the software kind — the mechanical kind. My six-axis arm dropped a payload because the control loop stuttered at exactly the wrong moment. The GC paused for 47ms. The arm dropped 300 grams of PLA-and-steel end effector onto a glass build plate. Everything shattered.

That was February. By March I'd made a decision: the entire real-time control stack was coming out of Go. The CLI tools, the web dashboard, the log aggregator — those could stay. But anything running inside the control loop's hot path had to go.

The Stack, Before [what I nuked]

The original architecture was pragmatic for a prototype: Go for orchestration, Python for CV inference, C for the motor drivers via cgo. It worked for six months. Then I added real-time pose estimation. Then I added impedance control. Then the GC stopped being a background hum and started being a liability.

hotpath.go — before 
// Before — Go hot path
func computeInverseKinematics(target JointAngles) (JointAngles, error) {
    buf := make([]float64, 100_000) // allocation every call
    defer pool.Put(buf)
    // ... math the GC would interrupt ...
}

The allocation per call wasn't the problem. The problem was that Go's GC, even with GOGC=off and sync.Pool everywhere, still ran a non-trivial sweep phase. On a Pi 5 with the arm moving, that sweep happened every 200-300ms. We measured P99 control latency at 38ms. The target was 4ms.

"

You can't fix a garbage collector problem with more garbage collection tricks. You fix it by removing the GC.

— Me, at 2am, covered in filament dust

Rust Adoption, Week by Week [the messy truth]

Week 1 — Reading the ecosystem

The first week was pure reading. The embedded-hal ecosystem, real-time guarantees via cortex-m-rt, the ownership model for shared-memory motor state. I'd written Rust before — side projects, CLI tools — but never for anything with physical consequences.

Week 2 — Porting the IK solver

Week two, I ported the IK solver. 1,200 lines of Go became 980 lines of Rust with no unsafe. The borrow checker caught two data races I didn't know I had in the Go version. That hurt. It also saved me from what would have been a very expensive field failure.

ik_solver.rs — after 
// After — Rust hot path (zero alloc in inner loop)
fn solve_ik(
    target: &JointAngles,
    state: &MotorState,
    // stack-allocated working buffer
    workspace: &mut [f64; 4096],
) -> Result<JointAngles, IKError> {
    // No alloc. No GC. No surprises.
    let solution = iterative_solve(target, state, workspace)?;
    Ok(solution)
}
"

I shipped the Rust rewrite to production on a Friday. Saturday morning I ran a 12-hour stress test. Zero pauses. Zero dropped frames. 4.2ms P99. I sat on the workshop floor for five minutes staring at the oscilloscope.

The Numbers [because you asked]

Performance metrics

Here's what 11 weeks of porting bought:

  • P99 control latency: 38ms / 4.2ms — 89% reduction
  • Memory usage: 62% drop (the GC bookkeeping alone was 30MB on a 256MB budget)
  • Lines of code: 4,200 Go / 3,100 Rust (incl. 400 lines of unsafe for the GPIO framebuffer)
  • Regressions caught at compile time: 5 — two data races, three lifetime violations that would have been use-after-frees

Latency histogram

What I'd Do Differently [and what I'd do again]

The FFI lesson

The cgo bridge was the hardest part. Go calling Rust via C ABI works, but the ceremony is fragile. Two production incidents — one string-lifetime bug, one callback deadlock — taught me to wrap the FFI surface in a small C library with a well-tested protocol buffer interface. Next time I'd start with uniffi-rs.

The borrow checker was the real win

What I'd do again: everything else. The borrow checker isn't the enemy — it's the reviewer you wish you had. And running code that physically moves things? Rust's guarantees aren't optional there. They're the difference between a clean stop and a shattered print plate.

"

The arm doesn't know or care what language its controller is written in. But I do. And so does the floor around my printer.

— Workshop wisdom, Day 78
